# Rachel 论文介绍页套件（中文）

本文档整理了一套适合论文介绍页、项目主页和研究展示页面使用的中文文案，包括摘要、介绍页主文和图注说明。整体风格兼顾学术可读性与审稿语境，尽量避免宣传式表述，同时保留系统设计上的核心信息。

---

## 摘要

多步逆合成规划的关键难点在于，系统不仅需要为目标分子提出局部可行的拆分候选，还需要在跨步骤层面维持骨架一致性、官能团兼容性、路线收敛性与前体可执行性。近年来，大语言模型为逆合成任务带来了更强的表达与策略组织能力，但多数方法仍倾向于将路线规划视为一次性的文本生成问题，从而使中间决策过程难以追踪、失败原因难以定位、局部修正难以累积。Rachel 针对这一问题，将多步逆合成形式化为一个显式的结构化流程，即 `state -> action -> validation -> commit`。在这一框架下，系统首先维护当前路线状态，再生成候选断键或官能团变换操作，并将候选步骤放入沙箱进行局部验证，只有通过门控检查后才被写入主路线树。Rachel 采用分层设计：编排层负责会话持久化、节点调度与路线树管理，化学工具层负责分子分析、模板扫描、断键执行、复杂度评分和前向验证，LLM 则作为策略层参与候选比较、排序与解释，而非直接充当未经约束的化学知识源。除了一般性的前向可行性验证外，Rachel 还引入位点锚定审计，用于识别同骨架手写前体在取代位点上的潜在漂移错误，从而提升复杂杂环和药物样分子场景下的可审计性。总体而言，Rachel 的核心贡献并不在于简单地让 LLM 参与逆合成，而在于将其嵌入一个可回放、可验证、可恢复的多步决策工作流中，使路线规划从一次性答案生成转变为结构化的化学推理过程。

**关键词**：多步逆合成、化学推理、大语言模型、结构化工作流、前向验证、位点审计

## 介绍页主文

### 1. 背景与问题定义

逆合成规划的目标并不是为目标分子随意给出若干看似合理的前体，而是在一系列相互依赖的步骤中，持续维持化学上的真实性、结构上的一致性和路线上的可执行性。对于真实的多步合成问题来说，单步看似正确的选择，往往可能在后续步骤中引入错误的骨架变化、过早暴露的高反应性把手、未处理的官能团冲突，或者明显欠缺收敛性的线性路线。也正因如此，多步逆合成不能被简单地理解为若干局部断键建议的拼接，而应被视为一个跨步骤的动态决策问题。

近年来，LLM 在路线建议、候选解释和合成策略组织方面表现出一定潜力，但许多现有方法仍然将逆合成任务压缩为一次性文本生成：系统读取目标分子，随后直接输出一条或多条完整路线。这类做法在展示上较为直观，但其局限也同样明显。首先，中间判断过程通常不会被显式记录，导致结果难以审计；其次，错误往往以“整条路线不理想”的形式暴露，而不是以某个具体决策的局部失败被定位；再次，当用户希望修正局部步骤或比较多个竞争方案时，系统缺乏一种天然的状态承载机制来保留已有进展。

Rachel 的出发点正是重新定义这一过程。它不把逆合成视为一次性产出答案的生成任务，而是把路线规划形式化为一套结构化工作流：当前状态决定候选动作，候选动作先经过验证，再被选择性写入主路线树。这样，多步逆合成就从“给出路线”转变为“组织决策过程”。系统输出的不只是一个终态答案，而是一条由状态、候选、验证、拒绝与提交共同构成的可检查决策轨迹。

### 2. 项目定位

Rachel 是一个面向多步逆合成的形式化化学推理系统。它的核心目标不是追求语言层面的路线描述能力，而是将真实的路线规划过程组织为显式、可恢复、可审计的结构化对象。在 Rachel 中，LLM 的职责主要位于策略层：它负责读取压缩后的上下文、比较候选方案、做出选择并生成解释；而与化学事实直接相关的部分，例如分子分析、官能团识别、反应模板扫描、断键执行、前向可行性验证和兼容性检查，则尽量下沉到工具层实现。

这种设计带来两个直接后果。第一，系统中的每一步都可以被定位到明确的数据结构上，而不是只存在于自然语言文本中。第二，LLM 的作用被约束在“决策组织”而不是“直接决定化学事实”上，这使得系统更容易积累失败经验、更容易比较替代路线，也更容易在研究环境中复现实验过程。

从工程形态上看，Rachel 当前仍是一个活跃的研究代码库。仓库中同时包含主流程代码、化学工具、实验数据、基准脚本、可视化输出和论文准备材料。这意味着它已经具备较完整的方法雏形与实验支撑，但当前重点仍是研究迭代与方法验证，而非轻量化的软件分发。

### 3. 系统结构

Rachel 的系统结构可以概括为三层。

第一层是编排与状态管理层。该层负责维护 JSON 会话、调度当前待处理分子、管理沙箱候选、记录决策历史，并将已接受的步骤写入合成树。在实现上，这一层通过命令式协议组织系统行为，例如 `init`、`next`、`explore`、`try_*`、`commit`、`finalize`、`report` 与 `export`。这一协议并不是为了模拟终端交互本身，而是为了让多步规划过程具有明确的状态边界与可恢复性。

第二层是化学工具层。它提供路线规划所需的化学计算能力，包括分子静态分析、官能团检测、可断键位点识别、反应模板扫描、断键与 FGI 执行、前向验证、复杂度评分、保护基需求分析与兼容性检查。与许多仅把工具层当作模板检索器的系统不同，Rachel 更强调工具层的门控作用：模板命中本身并不足以推动路线前进，真正的推进仍要依赖多维验证结果。

第三层是 LLM 策略层。Rachel 并不要求 LLM 从头生成整个路线，而是让其在结构化上下文中进行候选比较与策略判断。为了避免底层 SMARTS 命中和局部结构信息过度膨胀上下文，系统默认提供压缩视图，仅暴露足够支持决策的分子概要、官能团摘要、候选断键位点、FGI 候选和兼容性警告。当需要进行更细粒度检查时，系统再通过 `explore` 等接口按需展开局部信息。

### 4. 核心工作流

Rachel 的主流程可以概括为以下几个阶段。

首先，系统通过 `init` 创建一个新的规划会话。此时会建立目标分子的路线树根节点、初始化 BFS 队列，并根据目标复杂度设置相应的规划状态。与一次性输出不同，Rachel 从一开始就把路线规划视为可持续更新的会话对象。

其次，系统通过 `next` 选择当前待处理分子，并为其生成压缩的决策上下文。该上下文通常包括分子基本属性、复杂度评分、功能团摘要、可断键位点和可用的 FGI 选项。对于简单到可以直接接受为 terminal 的分子，系统也支持快速通道；而对于需要继续拆分的分子，则进入标准决策流程。

随后，LLM 或研究者可以通过 `explore`、`explore_fgi` 查看某个候选位点的更完整信息，也可以通过 `try_bond`、`try_fgi` 或 `try_precursors` 将特定方案放入沙箱。这里的关键原则是：候选先在局部环境中验证，而不是立即写入主路线树。这样，多个竞争方案可以在同一状态下被比较，而不会破坏当前主树结构。

在沙箱阶段，系统会对候选前体进行一系列检查，包括复杂度信息、前向可行性评估、原子守恒、骨架对齐、官能团兼容性以及必要时的位点审计。若候选步骤通过门控，系统才允许执行 `commit`，将反应节点与前体节点正式写入合成树，并把新的待处理分子加入队列。若候选未通过，则其失败原因会被保留为审计记录，而不是简单被覆盖。

最后，当所有叶节点都被接受为 terminal 或 advanced terminal 后，系统可以执行 `finalize`、`report` 和 `export`。此时会生成路线树文本、前向报告、起始原料列表、可视化 JSON、HTML/Markdown 报告以及完整的会话快照。这种导出方式不仅服务于展示，也使得后续实验复查与人工修订具有统一的数据入口。

### 5. 关键机制与技术特点

Rachel 最重要的特征之一，是其“沙箱优先”的工作方式。系统中的候选步骤并不直接进入主路线，而是先经过局部验证和比较。这一设计从根本上改变了 LLM 在逆合成中的使用方式：模型不需要一次性给出不可撤销的路线，而是能够在显式状态下探索多个局部备选，并以较低代价回退或替换。

第二个关键特征，是前向验证被视为硬门，而不是辅助分数。Rachel 的验证器并不会因为候选方案看起来“有模板支撑”就自动放行。相反，系统会检查原子守恒与骨架平衡、尝试模板的正向执行、评估最大公共子结构对齐情况、分析键变化是否符合反应类型预期，并检查官能团是否与设定反应条件冲突。这使得 Rachel 更接近一个以验证为核心的规划系统，而不是单纯的候选扩展器。

第三个值得强调的机制，是位点锚定审计。在复杂杂环和同骨架修饰问题中，一个前体即使与目标共享相同核心骨架，也可能在关键位点上发生错误替换。这类错误往往不会被一般性骨架匹配直接识别。Rachel 通过比较主要前体与目标在相同骨架上的局部取代位点，显式判断“变化是否发生在正确的位置”，从而避免一些在药物样分子场景中非常常见但也非常隐蔽的假阳性路线。

第四个特征，是系统对“决策上下文”进行了明确分层。Rachel 并不默认把底层检测结果完整暴露给 LLM，而是提供 `compact`、`full`、`status`、`tree` 等不同粒度的视图。这样做既减少了上下文冗余，也使系统能够把“局部审计”和“全局决策”区分开来。对于需要扩展到更长路线、更多候选的情形，这种分层上下文是维持稳定性的必要条件。

### 6. 当前仓库状态与方法边界

需要说明的是，Rachel 当前更适合作为研究方法系统来理解，而不是一个已经完成产品化打磨的公共软件包。仓库内包含大量实验数据、阶段性结果、评测脚本和研究笔记，这反映出系统正在持续演化之中。它已经具备完整的主链条能力，也已经可以支持路线生成、验证和可视化展示，但在安装体验、接口稳定性和仓库轻量化方面，仍保留明显的研究环境特征。

这一边界并不削弱 Rachel 的方法价值，反而更准确地体现了其当前用途：它首先是一个用于研究和验证“结构化逆合成推理”这一问题设定的系统。也正因为如此，Rachel 的重点并不是包装为尽可能简洁的调用接口，而是把推理过程中的关键环节尽量显式化，包括状态、候选、失败、审计和导出结果。

### 7. 总结

Rachel 所提出的，并不仅仅是一个由 LLM 驱动的逆合成程序，而是一种组织逆合成推理的方式。它通过显式状态、沙箱候选、前向验证、位点审计、路线树记忆和会话持久化，将多步路线规划从一次性答案生成转化为可检查、可恢复、可比较的结构化决策流程。对于真实化学规划而言，这种从“输出答案”到“组织过程”的转变，可能比单一步骤命中率的提升更有意义，因为它直接决定了系统是否能够在复杂目标、多方案竞争和持续修正的环境中保持可用性。
