# Rachel：一种面向多步逆合成的形式化化学推理系统

## 摘要

多步逆合成并非单纯的前体生成任务，而是一个要求系统在多轮状态更新中持续维持化学合法性、决策可解释性与过程可审计性的推理问题。现有模板方法虽然具有较强的可解释性，却缺乏灵活的高层策略组织能力；现有基于大语言模型的方法虽然具备丰富的知识先验，却往往将关键决策隐藏在自由文本中，缺少显式动作空间、验证闭环与失败治理机制。为此，我们提出 Rachel——一种面向多步逆合成的形式化化学推理系统。Rachel 将逆合成建模为一个状态—动作—验证—提交的闭环过程：分子、队列与路线树共同构成推理状态，断键、官能团互变（functional group interconversion, FGI）与自提前体构成合法化学算子，沙盒（sandbox）试探用于局部比较、回退与选择，正向验证、原子守恒、骨架对齐与官能团兼容性检查共同充当验证器，而模板证据、LLM 决策理由、验证记录与失败记录共同形成可审计的推理痕迹。系统进一步通过 JSON 命令接口与会话持久化机制，将 LLM 从自由生成器转化为结构化化学动作空间中的策略决策层。【实验回填占位｜EXP-ABS-01：此处在全量实验冻结后回填摘要中的经验结果总结，建议按“工具层主结果 → 多步主结果 → 失败治理主结果”三句式压缩撰写；具体写法见 `E:\Python\skills\Rachel\plan\实验回填模板.md`。】总体而言，Rachel 说明，多步逆合成的关键不只是生成一条路线，而是把路线生成过程提升为一个可验证、可解释、可追溯的化学推理对象。

## 引言

逆合成分析是有机合成规划中的核心任务，其目标是在给定目标分子的前提下，逐步回溯出一条由可得原料通向目标产物的合成路径。过去几十年中，计算机辅助逆合成系统围绕模板规则、反应数据库、启发式搜索与学习型模型持续发展，使得单步前体预测与局部反应建议取得了显著进展。然而，与单步预测相比，多步逆合成仍然更接近一个复杂的推理过程：系统不仅需要提出候选前体，还需要在多轮状态更新中管理分支、控制步数、处理保护/脱保护、协调官能团互变与关键断键，并持续判断每一步在化学上是否成立。换言之，多步逆合成的本质并不只是“生成一个答案”，而是“组织一个可信的化学推理过程”。

现有方法在这一点上仍存在明显缺口。模板方法具有良好的可解释性与可控性，但往往受限于覆盖范围、模板优先级与组合爆炸，难以承担复杂多步规划中的高层策略组织工作。神经网络与数据驱动方法在候选生成方面更具灵活性，却常常以牺牲过程透明度为代价：中间状态难以显式表示，错误难以定位，失败难以治理。近年来，大语言模型为化学规划带来了新的可能性，因为它们在高层策略组织、语义归纳与知识调用方面表现出显著优势；但如果将逆合成直接交由自然语言自由生成，关键决策就会被隐藏在不可检查的文本中，系统也缺少统一的动作语义、状态恢复机制和验证闭环。这种范式尤其不适合多步任务，因为多步规划的真正困难恰恰来自过程控制，而不是一次性答案表达。

我们认为，多步逆合成更适合被理解为一种**形式化化学推理**问题。所谓“形式化”，并不是将合成路线严格编码为逻辑定理并交由证明器验证，而是将多步规划过程中的关键组成——状态、动作、检查器与证据——显式化、对象化和协议化。在这一视角下，当前待拆解分子、待处理队列、已有路线树与策略记忆共同构成推理状态；断键、官能团互变（functional group interconversion, FGI）、保护/脱保护以及自提前体建议构成合法动作；原子守恒、正向验证、骨架一致性与官能团兼容性检查共同充当验证器；而模板证据、步骤记录、LLM 选择理由与失败日志则形成可审计的推理痕迹。这样的系统并不要求模型“凭空写出”路线，而是要求模型在一个受约束、可检查的化学动作系统中持续做出策略决策。

基于这一观点，本文提出 **Rachel：一种面向多步逆合成的形式化化学推理系统**。Rachel 通过结构化 JSON 命令接口、会话持久化、BFS 驱动的多步编排、模板/FGI/自提前体三类动作、沙盒（sandbox）试探机制以及验证门（validation gate），将 LLM 的高层策略能力与化学工具层的事实计算能力统一在同一会话框架中。与将 LLM 作为自由答案生成器的做法不同，Rachel 将 LLM 放置在策略层：LLM 负责阅读结构化上下文、比较候选方案、决定下一步动作，并在失败后调整策略；化学工具层负责枚举可断键位、执行 FGI、生成前体、分析分子属性并进行验证；验证门则负责判断某一步是否满足基本化学约束。通过将“试探”与“提交”严格分离，Rachel 允许系统在不污染主路线树的情况下比较多种局部方案，从而将多步逆合成转化为一种更接近科学实验设计与受控推理的过程。

这一形式化视角带来了两个直接收益。第一，它显著提升了多步规划过程的可审计性：系统输出的不再只是最终路线，还包括每一步的证据对象、验证记录、失败原因与策略痕迹。第二，它为评测目标提供了更清晰的分层：对于真实多步规划而言，“是否重建出参考路线”只是一个维度，“是否得到一条化学上可接受且过程可解释的路线”则是另一个同样重要、甚至更接近实际使用场景的维度。我们因此主张，在多步逆合成中应区分参考路线相似度（`reference-route similarity`）与实用成功（`practical success`），并围绕这两个维度组织系统评测与结果解释。

## 相关工作

近年来，逆合成研究大致沿着四条主线持续演进：一是以模板与启发式搜索为核心的经典计算机辅助合成规划方法；二是以模板自由预测、图编辑或序列建模为代表的单步数据驱动方法；三是围绕路线质量、多样性、收敛性和完备性展开的多步规划与评测方法；四是近两年迅速发展的、以大语言模型和智能体（agent）系统为中心的新一代化学规划框架。Rachel 与这些方向均存在直接关联，但其关注点并不完全等同于任何单一已有范式：它既不满足于单步前体预测，也不把多步规划简化为裸露的搜索算法，更不把语言模型当作自由文本生成器，而是试图将这些能力统一到一个形式化化学推理协议之中。

### 1. 模板搜索与经典多步规划

经典多步逆合成系统通常建立在“单步候选生成 + 树搜索”这一基本范式之上。近年来，以 EG-MCTS（2023）和 MEEA*（2024）为代表的方法进一步推进了该方向的发展：前者通过经验引导网络增强 Monte Carlo tree search 的扩展质量，后者则将 A* 的启发式扩展与 MCTS 的探索行为相结合，以改善复杂多步规划中的搜索效率与成功率。这类方法的重要贡献在于，它们把多步逆合成明确地表述为一个搜索问题，并围绕搜索策略、估值函数与预算利用展开系统优化。与此同时，诸如 Diversity & Completeness（2025）和以 DFPN 家族为代表的工作，则进一步指出：成功率并不是多步规划的唯一目标，路线多样性、覆盖度和完备性同样构成算法设计的重要维度。

这一路线与 Rachel 有显著交集，因为 Rachel 同样强调多步推理中的状态组织与搜索控制；但两者的切入点有所不同。传统搜索方法的重点通常在于如何在既定单步扩展器基础上提升整树求解效率，而 Rachel 进一步关注**动作语义、局部试探、验证门和审计痕迹**是否被显式纳入系统协议。换言之，EG-MCTS、MEEA* 等方法主要回答“如何更有效地搜索”，而 Rachel 试图进一步回答“系统在搜索的每一步究竟做了什么、为什么做、如何被验证以及失败如何被记录”。

### 2. 模板生成与模板自由单步预测

另一条重要主线是围绕单步逆合成预测展开的模板生成与模板自由方法。以 Site-Specific Template Generative（2024）为代表的工作强调在特定位点上生成反应模板，并结合可行性评分器提升位点可控性与模板解释性；这类方法的贡献在于并不完全放弃模板，而是试图让模板的生成与选择更加动态。与之并行的另一方向则是模板自由学习方法，例如 UAlign（2024）通过无监督 SMILES 对齐强化不变子结构复用，Node-Aligned Graph-to-Graph（2024）通过显式节点对齐直接学习产物到反应物图变换。这类方法在 USPTO 等标准单步基准上显著推进了 top-k 表现，也说明单步任务中的结构表示与对齐机制仍然是性能上限的重要决定因素。

然而，从 Rachel 的视角看，这一类工作大多聚焦于“如何更准确地完成一步预测”，而较少涉及多步过程中状态维护、失败回退与策略治理的问题。即便单步预测精度足够高，多步规划仍然需要回答诸如“当前是否应优先断键还是先做 FGI”“当前候选是否足以进入全局路线树”“失败是否应该成为后续决策的上下文”之类的问题。因此，Rachel 并不把单步方法视为竞争性替代品，而更接近于把它们视作**可嵌入形式化推理框架的局部算子或候选生成器**。这一点也是 Rachel 与“单步精度驱动”范式之间的根本区别。

### 3. 路线质量、收敛规划与评测口径扩展

随着多步规划系统逐渐成熟，研究重点也从“能否找到一条路线”转向“找到的路线在什么意义上是好的”。例如，Convergent Planning（2025）将收敛式路线设计引入逆合成规划，强调共享中间体与路线开发效率；Diversity & Completeness（2025）则指出，只追求单一路线成功率会掩盖搜索完备性与路线多样性的重要性；RetroScore（2025）等工作进一步尝试为候选路线建立更细粒度的打分标准。这些工作共同推动了逆合成评测从单一的 solve/not-solve 口径走向多目标、多维度的质量评价框架。

Rachel 在这一点上与上述工作高度一致，但其进一步推进之处在于：它不仅关注路线“结果”的质量，还关注路线“生成过程”的质量。换言之，Rachel 不只问路线是否短、是否收敛、是否多样，还问这些路线是通过何种动作被逐步构造出来的、每一步是否通过了验证、哪些失败曾被尝试过、最终路线是否有足够的证据支撑。因此，Rachel 所主张的 `practical success` 与 `reference-route similarity` 分离评价，既受到近年来路线评测扩展趋势的启发，也试图在此基础上加入**过程可验证性**这一尚未被充分制度化的维度。

### 4. LLM、智能体与人机协同逆合成

近两年，基于大语言模型的逆合成与化学智能体（agent）系统迅速成为热点。该方向的代表工作包括 DeepRetro、SynAsk、ChemCRAFT、RetroInText、Llamole 等。它们从不同角度展示了语言模型在化学规划中的潜力：有的强调多轮推理与重试，有的强调约束型决策与多代理协同，有的强调多模态表示与 in-context route 建模，也有工作将 agentic reinforcement learning 与工具调用能力结合起来，用以增强模型在复杂化学任务中的执行能力。总体而言，这些工作共同说明：LLM 的真正价值往往不体现在“背诵一个反应模板”，而体现在高层策略组织、任务分解、条件约束理解和跨步骤信息整合。

然而，现有 LLM/智能体方向也暴露出一个共同问题：如果系统仅仅把语言模型包裹在“反复生成—检查—重试”的外循环中，那么其核心决策依然可能停留在缺乏统一语义的文本层。此时，虽然系统具有某种程度上的交互性与灵活性，但动作对象、失败对象与验证对象之间仍未形成稳定的协议关系。Rachel 与这一方向最本质的不同之处在于，它并不把 LLM 首先理解为“更强的答案生成器”或“可自由试错的代理”，而是把 LLM 安置在一个**结构化化学动作空间中的策略层**。在 Rachel 中，工具调用不是附加功能，而是推理协议的一部分；验证器不是事后打分器，而是正式提交之前的门控环节；失败不是需要被隐藏的异常，而是可被记录、总结并回灌到后续决策中的系统对象。

### 5. Rachel 的位置与区别

综合来看，现有工作大致解决了四类不同问题：模板与搜索方法回答“如何高效扩展树”；模板生成与模板自由方法回答“如何提升单步候选质量”；路线打分与收敛规划工作回答“如何更全面地评价路线”；LLM 与智能体系统回答“如何增强高层策略与交互能力”。Rachel 的位置恰好处于这四条主线的交汇处，但它的目标不是简单拼接这些已有模块，而是提出一个更强的统一表述：**多步逆合成应被视作一个形式化化学推理系统，而不是模板枚举、单步预测或自由文本生成的简单叠加。**

因此，Rachel 的差异性主要体现在三个方面。首先，在问题表述上，Rachel 将多步规划明确为状态—动作—验证—提交的闭环过程，而不是单纯的产物到前体（product-to-precursor）映射或黑箱树搜索。其次，在系统实现上，Rachel 通过命令式会话、沙盒（sandbox）、验证门（validation gate）、路线树和审计状态，把“做了什么”“为什么这么做”“这一步是否通过验证”“失败发生在哪里”全部对象化。最后，在评测观念上，Rachel 明确区分参考路线相似度（`reference-route similarity`）与实用成功（`practical success`），并主张将过程可验证性纳入多步逆合成系统的核心评价维度。正是在这个意义上，Rachel 既继承了现有工作的有效成分，也试图为多步逆合成提供一个更适合人机协同、失败治理与系统审计的新范式。

## 问题定义

为了使多步逆合成从“答案生成”转化为“过程推理”，我们将其定义为一个**形式化化学推理任务**。给定目标分子 $m_{target}$，系统的目标不再只是输出一组候选前体，而是在一系列受约束的化学动作之上，逐步构造出一棵从目标分子回溯到可接受起始原料的路线树，并为每一步提供可检查的化学证据与决策痕迹。与传统将逆合成看作单步映射 $product \rightarrow precursors$ 的表述相比，这一定义强调多步任务中的状态演化、动作选择、局部验证与全局记录。

更具体地，Rachel 将多步逆合成中的任意时刻表示为一个推理状态
$S_t = (Q_t, T_t, A_t, C_t)$。其中，$Q_t$ 表示当前待展开分子的队列，$T_t$ 表示已生成的逆合成路线树，$A_t$ 表示审计状态与历史记忆，$C_t$ 表示当前分子的局部决策上下文。这样的表述有两个直接好处。第一，它承认多步逆合成本质上是一个**状态依赖任务**：下一步动作不仅取决于当前分子的结构，也取决于已经做出的历史选择、已形成的分支结构以及剩余预算。第二，它将会话中真正有价值的推理信息对象化，使系统能够在中断后恢复，也使实验评测能够从“结果对不对”进一步过渡到“过程如何形成”。

在该状态空间上，Rachel 定义一组具有明确语义的合法化学动作 $\mathcal{A}$。这些动作至少包括三类：其一是基于反应模板或规则的**断键动作**，用于将目标分子拆解为两个或多个前体片段；其二是**官能团互变动作**（FGI），用于在不直接断裂主骨架的前提下改写目标分子的反应可达性；其三是**自提前体动作**，即由 LLM 基于结构化上下文提出前体，再由系统进行验证。除此之外，系统还允许 `accept-terminal` 与 `skip` 等控制性动作，用于在多步搜索中显式处理终止与放弃情形。这里的关键并不在于动作数量的多少，而在于每个动作都有明确的输入、输出与后果，不再是难以约束的自由文本操作。

为了避免将“候选生成”误认为“有效推理”，Rachel 进一步为每个动作引入局部试探与正式提交的区分。给定当前状态 $S_t$ 和候选动作 $a \in \mathcal{A}$，系统首先执行一个局部试探过程
$\mathrm{try}(S_t, a) \rightarrow r_t$，其中返回结果 $r_t$ 包含候选前体、正向验证结果、原子平衡分析、潜在冲突与失败原因。只有当试探结果满足基本化学约束，且策略层决定接受该动作时，系统才执行正式提交
$\mathrm{commit}(S_t, a) \rightarrow S_{t+1}$，将该步写入路线树并更新队列与审计状态。通过这一“先试探、后提交”的两阶段机制，Rachel 将多步逆合成中的局部比较、回退和选择显式化，从而避免未经验证的候选直接污染全局路线结构。

在验证层面，Rachel 将化学合法性视为一类一等约束，而非后处理补丁。对于任意候选动作，系统并不只检查其是否产出了某组前体，而是要求这些前体经过一组验证器的共同检验。验证器包括但不限于：原子守恒分析、正向反应验证、骨架一致性检查、键变化拓扑合理性以及官能团兼容性分析。换言之，在 Rachel 中，“这一步看起来像一个化学步骤”并不足够；系统必须回答“这一步在何种意义上通过了哪些检查”。因此，多步逆合成的输出也不再仅仅是路线本身，而是一组伴随路线共同生成的**证据对象**。

基于上述定义，Rachel 所追求的目标也与传统逆合成系统略有不同。我们并不将系统目标简单设定为最大化与某条参考路线的重合度，而是将其拆分为两个互补维度。第一是**参考路线相似度（reference-route similarity）**，即系统路线与参考路线在关键断键、步骤数或边结构上的一致程度；第二是**实用成功（practical success）**，即系统是否能在既定预算下产出一条化学上可接受、起始原料可接受、关键步骤可验证且过程可解释的路线。前者衡量系统对既有数据模式的贴近程度，后者更接近真实合成规划的使用目标。Rachel 的问题定义正是围绕这两个维度展开：不是仅仅重建已有答案，而是在可验证约束下组织一条可信的化学推理过程。

从这个角度看，Rachel 与形式化推理系统之间具有明确的结构同构关系。当前待展开分子及其上下文相当于证明状态（proof state），断键、FGI 与自提前体对应于可选择的策略动作（tactics），局部试探相当于策略执行（tactic application）的中间检查，正向验证与化学守恒约束相当于检查器（checker），而路线树、模板证据、决策理由与失败记录共同构成推理痕迹（proof trace）。我们并不声称 Rachel 是严格意义上的化学证明器；但我们强调，Rachel 已经将多步逆合成从隐式自然语言流程推进为一种**可协议化、可审计、可验证的形式化化学推理过程**。这一定义构成了全文方法设计与实验评测的理论基础。

## 方法总览

在上述问题定义之下，Rachel 的设计目标并不是简单堆叠“LLM + 化学工具”，而是构造一个由**状态管理、动作编排、局部试探、验证门与证据沉淀**共同组成的推理框架。整体上，Rachel 由四个相互配合的层级构成：其一是面向多步会话的命令式交互层，用于将 LLM 的决策组织成显式动作协议；其二是面向当前分子的决策上下文生成层，用于把分子结构编译为可供策略选择的化学动作空间；其三是面向动作执行的化学工具层，用于执行断键、FGI、前体生成与验证；其四是面向全局过程控制的树与审计层，用于维护路线结构、记录历史选择并沉淀失败经验。通过这四层的配合，Rachel 将“提出路线”转化为“在一个受约束系统内逐步完成推理”。

### 1. 命令式会话与状态持久化

Rachel 的最外层接口是一个结构化命令协议，而不是单次 prompt 驱动的自由对话。系统通过 `init`、`next`、`explore`、`try_*`、`commit`、`accept`、`skip`、`finalize` 等命令组织多步交互，使每一步决策都有明确的动作语义与状态后果。这样的设计使得 LLM 不再扮演“直接输出整条路线”的角色，而是以策略决策者的身份在固定协议中行动。与此同时，Rachel 将当前上下文、沙盒结果、路线树、审计状态与待处理队列统一持久化到会话文件中，从而将多步逆合成从“一次性的自然语言会话”转化为“可恢复、可回放、可分析的推理进程”。这一点对于多步任务尤其重要，因为真实多步规划很少在一次交互中完成，系统必须能够处理中断恢复、回顾历史与持续迭代。

### 2. 决策上下文生成：从分子到动作空间

Rachel 的核心不在于把原始分子字符串直接交给 LLM，而在于把当前分子**编译**为结构化决策上下文。给定一个目标分子，系统首先生成其描述符、官能团信息、复杂度评分、潜在可断键位、FGI 候选以及官能团兼容性警告。这个上下文既包含静态结构事实，也包含与决策直接相关的候选动作集合，从而把“你觉得怎么拆”转化为“在这些具备明确化学语义的候选动作中，下一步应选择什么”。这一层的价值在于将分子理解与动作组织前置：LLM 看到的不是一个未加工的化学对象，而是一个已经被投影到策略空间中的决策问题。

其中，断键候选的组织采用了带有化学先验的排序机制。系统根据键类型、是否处于环内以及匹配模板的置信度，对可断键位进行启发式排序，从而在有限的推理预算下优先暴露更可能有价值的断键位置。这构成了一种轻度的断键优先算子先验（disconnection-first operator prior）：它并不排除 FGI 或自提前体，而是试图在有限上下文中先组织最具代表性的候选。与此同时，FGI 选项被作为与断键并列的一类一级动作保留下来，以避免系统将逆合成错误地简化为纯断键问题。

### 3. 动作执行层：断键、FGI 与自提前体

在动作执行层，Rachel 允许三类主动作进入同一推理闭环。第一类是基于模板和规则的断键执行，用于产生片段化前体并保留与模板对应的证据；第二类是 FGI，用于在不直接拆分骨架的情况下改变目标分子的反应可达性，尤其适用于氧化还原、保护/脱保护以及其他单片段或局部变换场景；第三类是自提前体动作，用于允许 LLM 在必要时越过模板覆盖边界，直接提出可能的前体组合，再由系统验证其合理性。这样的设计体现了 Rachel 的基本原则：策略层可以灵活，但动作层必须具备明确语义与验证入口。

值得强调的是，Rachel 并未将 FGI 视为断键规划之后的附属修饰步骤，而是将其视为改变搜索空间结构的核心算子。在多步逆合成中，许多关键路线并不是先找到断键再补充变换，而是必须先通过官能团互变塑造出后续可拆解的中间体。Rachel 将 FGI 与断键共同纳入主动作空间，正是为了避免系统在表示层面过早固化为“所有合理路线都应以断键为起点”的狭义范式。

### 4. 沙盒（sandbox）机制：先比较，再承诺

Rachel 的另一关键设计是将动作试探与正式提交严格分离。对于任何候选动作，系统都允许先在沙盒（sandbox）中局部执行，生成前体、计算复杂度、进行正向验证与原子平衡检查，并返回潜在风险、冲突和失败原因。这一阶段不会修改全局路线树，因此策略层可以安全地比较多个动作后再选择其一。只有当某个候选被确认接受，系统才将其以正式步骤的形式写入树结构并更新待处理队列。这种设计使 Rachel 具备了“验证驱动试错”的能力：系统既能够探索，又不至于因一次局部错误而破坏整体推理轨迹。对于多步规划而言，这比一次性提交整条路线更符合实际化学推理的工作方式。

### 5. 验证门：将化学约束前置为系统规则

在 Rachel 中，验证不是一个附属打分器，而是动作闭环中的强约束环节。系统会对候选前体执行一组验证过程，包括原子守恒、正向反应检查、骨架一致性、键变化拓扑合理性以及官能团兼容性等。这些检查共同决定某个局部动作是否具备进入正式路线树的资格。这样的设计有两个重要意义。其一，它显式抑制了“文本上听起来合理、化学上却不成立”的候选。其二，它将失败从不可解释的黑箱现象转变为具名的结构化结果，例如原子不平衡、模板正向不支持、官能团冲突或候选不可解析等。换言之，Rachel 不是在生成之后补做打分，而是在推理过程中把化学合法性内化为系统规则。

### 6. 路线树与审计状态：把推理过程沉淀为对象

为了让多步逆合成的过程真正可分析，Rachel 将所有正式提交的步骤组织为一棵显式路线树，并通过 canonical SMILES 去重来处理重复分子与收敛分支。每个反应节点不仅记录产物与前体，还记录模板证据、动作类型、可能的 LLM 决策信息以及验证结果。与此同时，系统维护一个全局审计状态，用于保存近期决策、失败历史、保护基状态、线性步数目标和高层策略信息。路线树负责保存“系统做了什么”，审计状态负责保存“系统为什么这么做、哪里失败过、后续应警惕什么”。二者共同构成 Rachel 作为形式化化学推理系统的记忆机制。

### 7. 方法的整体意义

总体而言，Rachel 的方法总览可以概括为一句话：它不是把化学问题交给语言模型自由回答，而是把语言模型接入一个由状态、动作、验证与证据共同定义的化学推理协议中。LLM 的角色因此从“答案生成器”转变为“策略组织者”，化学工具从“外部插件”转变为“动作与检查的执行层”，而整个系统则从“给出建议的代理”升级为“可验证、可恢复、可审计的多步化学推理框架”。这一方法组织方式既构成了 Rachel 与现有自由生成范式的本质区别，也为后续实验部分关于 FGI 价值、多步 utility 与 failure governance 的分析提供了统一基础。

## 系统设计细节

如果说前一节回答的是“Rachel 的方法由哪些原则组成”，那么本节要回答的问题是：**Rachel 如何把这些原则落实为一个真正可运行的多步逆合成系统**。在实现层面，Rachel 采用分层架构：最上层是面向 LLM 的命令式交互与会话层，中间层是负责搜索控制与状态推进的编排层，底层是负责化学事实计算与动作执行的工具层，旁路则是路线树、审计状态、报告与导出模块。这一设计既保持了系统的工程可维护性，也使不同层级的职责边界足够清晰：LLM 不直接接触底层化学对象，化学工具不直接输出决策建议，编排器不硬编码合成策略，而是负责把状态、动作和验证组织为一套统一协议。

### 1. 总体架构与模块划分

Rachel 的主仓库结构可以概括为三部分。第一部分是 `main/`，承担交互与编排职责，包括 `retro_cmd.py` 命令接口、`retro_session.py` 会话持久化、`retro_orchestrator.py` 搜索编排器、`retro_tree.py` 路线树、`retro_state.py` 审计状态，以及报告和导出模块。第二部分是 `chem_tools/`，这是系统的化学计算底盘，包含分子分析、官能团识别、模板扫描、断键执行、正向验证、复杂度评分、官能团警告与智能封端（`smart_cap`）等模块。第三部分是 `tools/` 与 `tests/`，分别承担评测平台、批量脚本和实验验证功能。这样的结构设计使 Rachel 既是一个交互式多步系统，也是一个可独立测试、可批量评估的研究平台。

从接口职责来看，Rachel 特别强调“工具层只做事实输出，不做决策提示”。换言之，化学工具层负责返回结构化化学结果，但不直接生成自然语言建议；决策解释则由上层 LLM 在拿到结构化上下文后完成。这一职责分离使得系统能够同时兼顾两种需求：一方面，底层计算结果足够可重复、可单元测试；另一方面，上层策略层又保留了足够的灵活性，以处理多步规划中难以完全模板化的选择问题。

### 2. 命令式接口：将推理过程离散化为可执行协议

Rachel 的最外层接口是一个 JSON-in / JSON-out 的命令协议。系统支持 `init`、`next`、`context`、`explore`、`explore_fgi`、`try_bond`、`try_fgi`、`try_precursors`、`select`、`commit`、`accept`、`skip`、`tree`、`status`、`finalize`、`report`、`export` 等一组明确命令。与许多基于 prompt 的代理式系统不同，Rachel 并不把一次多步规划封装为一段长自然语言会话，而是将推理过程拆解为一串具有显式语义的操作。这种离散化设计有三点关键价值。

首先，它为多步逆合成提供了可重复的动作语义。一次断键探索、一次 FGI 试探或一次正式提交，都以统一的结构化命令表示，而不是依赖 prompt 模板中的松散措辞。其次，它天然支持程序化调度与批量实验，因为任何一步都可以通过标准化接口复现、记录和回放。最后，它降低了 prompt 漂移与会话遗忘带来的不稳定性：LLM 在 Rachel 中不是通过“记住刚才说过什么”来维持推理，而是通过“读取当前系统状态并执行下一个协议动作”来推进过程。

这种命令式接口本质上把 LLM 从传统意义上的“聊天对象”转化为一个在形式化动作空间中行动的策略控制器。LLM 的能力仍然重要，但它不再通过自由叙述直接控制系统，而是通过一套可检查、可记录的命令协议发挥作用。这也是 Rachel 得以实现可审计推理的重要基础。

### 3. 会话持久化与当前工作区

多步逆合成的一个核心工程问题，是如何保存中间状态。Rachel 的答案是：将完整推理过程映射到一个持久化会话对象中。会话文件保存目标分子、配置参数、当前工作区、沙盒（sandbox）区域、路线树、审计状态和待展开队列等内容。当前工作区中既包含当前分子的上下文分析，也包含键位摘要、FGI 摘要与临时试探结果；沙盒区域则专门用于保存尚未正式提交的候选尝试及其验证结果。

这种设计并不仅仅是为了“断线可恢复”。更重要的是，它改变了多步逆合成的交互范式：系统不再依赖模型对历史对话的记忆，而是将历史过程对象化为一个可读、可写、可恢复的状态文件。这样一来，复杂规划中的几类关键过程——局部比较、失败回顾、路径回溯、跨轮次继续——都可以在统一的状态表示下完成。这一点对于真实多步逆合成尤为重要，因为在复杂目标上，策略调整几乎不可避免，而没有持久化状态的系统很难支持这种渐进式推理。

### 4. 编排器：BFS 驱动的多步决策闭环

Rachel 的核心控制器是编排器。编排器负责从待处理队列中取出下一个分子、生成决策上下文、接收策略层给出的动作选择、调用工具层执行、进行正式提交并更新树与队列。当前实现采用 BFS 风格的展开策略：系统围绕待拆解分子队列逐步推进，优先确保路线树在广度上可持续扩展，而不是过早陷入单一路径的深度展开。这样的设计与 Rachel 对“可管理多步状态”的总体目标一致，因为广度优先更容易在有限预算下维护多分支可达性，并减少系统过早进入局部不可逆死路的风险。

在编排细节上，Rachel 为每个分子设置了两种决策层级。对于被复杂度阈值判定为可直接接受的简单分子，系统进入 quick-pass 通道，将其快速标记为 terminal；对于仍需展开的分子，系统则进入标准流程，生成完整决策上下文并等待进一步探索、试探与提交。通过这种机制，Rachel 将“无需推理的终止情形”与“需要局部搜索和策略比较的关键情形”分离开来，从而更高效地利用推理预算。

### 5. 决策上下文的分层表示

Rachel 生成的决策上下文并非单一固定格式，而是采用分层表示。默认情况下，系统返回 compact 视图：其中包括分子基本信息、复杂度、官能团、前若干个断键候选的概览、FGI 摘要、队列预览以及审计状态摘要。只有当策略层对某一键位或某类 FGI 候选产生兴趣时，才通过 `explore` 或 `explore_fgi` 请求进一步展开完整细节。这种设计一方面降低了多步对话中的 token 负担，另一方面也更符合真实推理过程：LLM 先对局部决策空间形成粗判断，再按需深入具体候选，而不是每一步都被迫阅读全量前体列表。

更重要的是，这种分层上下文使 Rachel 的状态表示更接近一个真正的推理工作区。compact 层负责维持全局感知与优先级判断，full 层负责支持局部论证与动作比较，status/tree 层负责支持更长程的过程管理与回顾。对于多步任务而言，这种上下文分层不是简单的工程优化，而是实现长期推理可持续性的关键手段。

### 6. 化学工具层：从结构分析到动作执行

Rachel 的化学工具层由多个功能明确的模块构成，其目标不是直接告诉系统“该怎么做”，而是提供可组合的化学事实与动作结果。分子分析模块用于提供分子式、重原子数、环信息、手性中心与描述符；官能团识别模块负责识别反应相关的局部结构模式；模板扫描模块负责从模板库中找到可适配的反应模板与可断键位；断键执行模块负责将指定键位转换为前体；FGI 执行模块负责执行单片段或局部变换；正向验证模块负责对候选前体进行化学合法性检查；复杂度模块则提供 CS score，用于辅助终止判定与多步预算管理。

这种模块化设计使 Rachel 能够把复杂的化学决策拆解为若干可测试、可独立演化的组成部分。例如，模板扫描关注“哪里可能可拆”，断键执行关注“拆了以后得到什么”，正向验证关注“得到的东西是否站得住”，复杂度模块关注“该不该继续拆”，而智能封端则弥补模板覆盖不足时的局部前体重构需求。通过将这些角色清晰划分，Rachel 使多步逆合成不再依赖某一个超级模型一次性完成全部推理，而是依赖多个职责明确的组件共同支撑一个统一推理协议。

### 7. 断键候选、FGI 候选与算子先验（operator prior）

Rachel 在候选组织层面有一个非常重要的实现特点：系统显式维护多类算子，但在排序上引入了轻度的算子先验（operator prior）。对于断键候选，系统将模板匹配置信度与键类型优先级组合为启发式分数，以此对可断键位进行排序。这意味着 Rachel 并非盲目枚举所有可能动作，而是在有限决策预算下优先暴露更有希望的断键位点。与此同时，FGI 候选则通过对 retro 模板的进一步筛选与执行，被保留为与断键并列的一类主动作，而不是作为断键失败后的纯后备补丁。

这个实现细节非常关键。它说明 Rachel 的搜索不是“模板枚举 + LLM 自由发挥”，而是一个同时包含动作多样性与预算控制的组织过程。系统承认断键在很多标准合成场景中具有较高优先级，因此采用了轻度的断键优先排序；但系统也承认大量有效路线必须先通过 FGI 改写反应可达性，因此保留了 FGI 作为一级算子。也正因为这种设计，Rachel 既能受益于断键动作在多数标准数据集上的高频结构先验，又能在实验中显式暴露断键优先偏置可能带来的系统性局限。

### 8. 智能封端（smart_cap）：模板与规则之间的桥梁

为了缓解“只靠模板执行仍然不够”的问题，Rachel 还引入了智能封端（smart_cap）模块。`smart_cap` 针对某个待断裂键位，分析两端原子的局部化学环境，包括元素类型、杂化状态、芳香性、邻近官能团等信息，再基于一组规则推断断键后各侧更合理的封端形式。最终，系统会返回若干带有反应类型说明、前体片段与置信度的候选方案。这一设计的意义，在于它在模板库和自由生成之间提供了一座桥梁：既保留了规则的可解释性，又允许系统在模板未完全覆盖的情形下，通过局部环境推理产生仍然具有化学意义的前体候选。

从系统角度看，`smart_cap` 还起到了一个非常实用的作用：它把“我知道这里该断，但我不确定前体具体该怎么写”的模糊状态，转化为“系统给出若干具备明确化学语义的前体候选，供策略层比较和试探”。这种从环境到前体形式的中间层，是 Rachel 区别于单纯模板执行器的重要实现细节之一。

### 9. 沙盒（sandbox）：将局部搜索显式化

Rachel 的沙盒（sandbox）不是普通日志缓冲区，而是一个真正的局部搜索工作区。每一次 `try_bond`、`try_fgi` 或 `try_precursors` 都会生成一个结构化的尝试结果，记录候选前体、复杂度分析、验证结果、原子守恒、潜在环路警告与相关反应信息。多个尝试会被顺序保存在沙盒中，策略层可以查看、比较、选择其一，也可以清空后重新探索。只有被显式选中的方案，才会在 `commit` 阶段进入正式路线树。

这一机制让 Rachel 的多步推理具备了“中间态可显式比较”的能力，这在许多基于自然语言直接输出答案的方法中是缺失的。具体来说，沙盒支持以下三种重要行为：一是比较多个局部候选而不污染主树；二是在失败后保留经验并继续探索；三是将“最终采纳的是哪个候选”明确记录下来。这使得 Rachel 的局部决策不再是不可追踪的一次性选择，而是一个具备可回顾证据链的局部搜索过程。

### 10. 正式提交：从局部动作到全局路线树

一旦某个候选在沙盒中被接受，Rachel 会执行正式提交。提交阶段不仅将反应写入路线树，还会更新前体节点的角色、复杂度与队列状态，并同步记录模板证据、LLM 决策信息、验证结果与潜在环路警告。对于新产生的前体，系统会进一步根据复杂度与可接受性标准判断其应被标为 terminal 还是 intermediate；若为 intermediate，则进入后续待展开队列。通过这一机制，Rachel 将局部动作与全局多步状态联系起来，使系统真正形成一个可逐步扩展的路线树，而不是一串彼此松散的局部建议。

值得注意的是，Rachel 的路线树使用 canonical SMILES 去重，这使得系统能够更自然地处理重复中间体与收敛分支。与简单的序列化路线不同，这样的双类型节点图结构更适合表达真实多步逆合成中的共享前体与汇聚逻辑，也为后续路线质量评估、失败复盘与可视化导出提供了良好基础。

### 11. 审计状态：让系统具备“过程记忆”

除正式路线树外，Rachel 还显式维护一个审计状态对象。它包含近期决策记录、失败尝试、保护基状态表、全局策略信息以及线性步数建议等内容。审计状态的作用并不是替代路线树，而是为系统提供另一类“过程记忆”：路线树告诉我们系统**做了什么**，而审计状态告诉我们系统**是如何一路走到这里的**。这对于多步任务尤其重要，因为复杂路线中的很多关键信息并不会完整地体现在最终树结构中，例如某个候选为何被拒绝、某个保护策略何时被规划、某类失败是否已经多次出现等。

通过将这些信息显式对象化，Rachel 使失败不再只是异常终止，而是系统性知识的一部分。换句话说，Rachel 的审计状态不仅服务于人类复盘，也服务于系统自身未来的策略调整与实验分析。

### 12. 报告、导出与离线复盘

作为一个研究系统，Rachel 的输出不仅是“最后得到什么路线”，还包括完整的复盘工件。系统能够导出会话文件、路线树、正向报告、可视化结果以及相关日志，从而支持离线审查、案例分析和实验批量归档。这些工件对于论文写作和系统评测都非常重要：一方面，它们使案例展示不再停留于挑选性的截图或口头解释；另一方面，它们也让失败 taxonomy、路线统计和多步 benchmark 可以建立在统一、可追溯的数据工件之上。

### 13. 本节小结

综合而言，Rachel 的系统设计细节体现出一种明确的工程哲学：**把每一个原本隐含在自然语言中的推理环节，尽可能转化为显式的对象、动作和记录**。命令接口负责把决策离散化，会话层负责把状态持久化，编排器负责把多步搜索组织化，工具层负责把化学事实结构化，沙盒负责把局部比较显式化，验证门负责把化学约束前置化，树与审计层则负责把推理痕迹对象化。正是这一整套细节设计，使 Rachel 不只是一个“能调用化学工具的 LLM 系统”，而成为一个更接近形式化化学推理框架的多步逆合成系统。

## 实验设置

为了验证 Rachel 作为形式化化学推理系统的有效性，我们并未将实验组织为单一榜单，而是采用一条由工具层、策略层、多步端到端、消融与失败分析共同组成的证据链。这样的设计对应本文的总体论证逻辑：首先要证明 Rachel 的化学动作空间本身是否具有稳定质量；其次要验证 LLM 在结构化上下文中是否真正提供策略增益；再次要检验系统在统一预算下是否能够完成具有实际意义的多步规划；最后则要说明系统失败是否可被分类、解释并纳入治理闭环。基于这一思路，本文实验分为 E1–E6 六个部分，分别对应工具层质量、LLM 协同、协同机制解释、多步端到端、必要消融以及失败模式与边界分析。

### 1. 实验总体组织与研究问题

本文的实验设置围绕三个核心研究问题展开。

**RQ1：Rachel 的结构化化学动作空间本身是否有效？**  
这一问题主要由 E1 回答。我们关注的不是 LLM 最终选了什么，而是当系统仅依赖模板、FGI 与底层化学执行层时，是否能够稳定产生与真实反应相关的候选前体。

**RQ2：当上下文被形式化之后，LLM 是否体现出额外的策略价值？**  
这一问题由 E2 和 E3 回答。我们不把 LLM 看成模板的简单替代，而是检验它在候选比较、动作选择、越模板边界与局部机制组织方面是否提供增益。

**RQ3：Rachel 是否能够在统一协议与统一预算下完成多步规划，并暴露出有意义的边界与失败模式？**  
这一问题由 E4、E5 和 E6 回答。这里的重点不只是“能否完成一条路线”，还包括路线是否化学上可接受、系统不同组件各自贡献多大，以及失败是否具有稳定的可解释结构。

这种分阶段设计还有一个方法学上的好处：它避免了把系统所有能力都压缩到一个单一数字中，从而更符合 Rachel 的设计哲学。由于 Rachel 本身强调状态、动作、验证与证据的分层显式化，因此实验也应当沿着这一分层结构逐层建立证据，而不是用一个粗粒度成功率掩盖系统内部不同模块的实际作用。

### 2. 数据集与实验资产

Rachel 的实验使用两类主要数据来源。第一类是单步反应数据，主要来自 USPTO50K 分片数据集，包括 `BTF1` 到 `BTF10` 以及其他相关文件。这一数据源主要用于评估工具层候选质量、单步模板/FGI 覆盖与 LLM 在结构化单步决策中的表现。第二类是多步路线数据，主要来自 PaRoutes v2 的 `n1` 和 `n5` 路线与步骤文件，用于构建多步评测池与多步端到端 benchmark。除此之外，项目还维护了标准化的实验目录结构，将核心测试、baseline、E1–E6 分阶段实验和结果分析统一组织到 `tests/` 目录中，从而保证单步、多步、失败分析和消融实验之间共享一致的工件规范。

对于单步实验，我们主要使用按反应类型拆分的 USPTO 子集进行大样本评估。这种设置的优点是可以同时观察总体质量与不同反应类型、不同难度分布下的局部表现，并减少由单一数据分布造成的偶然结论。对于多步实验，我们采用 PaRoutes 路线资产及其派生的数据池，以便同时支持静态横向可比评测与会话式主评轨评测。尤其是在 E4-v2 中，我们使用从 `n1` 与 `n5` 路线池中抽样构建的固定会话数据集，并对其进行 dev/eval 拆分，以确保多步实验中的目标集合、预算与统计口径在不同实验条件下保持一致。

从工件角度看，Rachel 的实验并不是只生成“一个分数”，而是会生成题集文件、提示文件、答案文件、逐条结果表、摘要统计、路线树、会话文件、报告和可视化页面等多个层级的输出。这种实验工件设计与 Rachel 的系统设计保持一致：既然系统强调可恢复与可审计，那么实验也必须建立在可复核的中间工件之上，而不是只保留最终汇总数字。

### 3. E1：工具层质量与稳定性

E1 对应本文的第一个 claim，即 Rachel 的模板 + FGI 化学工具层是否能够在不依赖上层策略干预的情况下稳定产生有化学意义的候选。该实验使用 USPTO50K 的 `BTF1`–`BTF10` 子集，通过 `run_baseline.py` 在大样本上系统评估断键候选、FGI 候选及两者合并后的覆盖表现。这里的核心不是证明“模板足够”，而是证明 Rachel 的动作空间不是无意义噪声，而是一个可作为策略层输入的高信息量候选集合。

E1 主要报告以下几类指标：`bond_exact_rate`、`combined_exact_rate`、`combined_partial_rate`、`bond_tanimoto_mean`、`combined_tanimoto_mean` 以及失败类别分布。之所以同时报告 exact、partial 与 Tanimoto，是因为它们对应不同层级的候选质量：exact 反映严格命中，partial 反映局部相关性，而 Tanimoto 分布则更适合揭示候选空间整体与真实前体的距离结构。结合失败类别分布，E1 还可用于识别系统性盲区，例如 FGI 缺口、模板误匹配和特定反应族的覆盖问题。对本文而言，E1 的角色是为后续所有“LLM 是否有策略增益”的结论提供前提：只有当底层动作空间本身足够有意义时，讨论上层决策才有价值。

### 4. E2 与 E3：LLM 协同增益与机制解释

E2 用于回答这样一个问题：当候选空间被结构化显式化之后，LLM 是否能够在其中做出比纯模板基线更有价值的选择。该实验基于 `run_llm_exam.py` 与 `run_isolated_blind.py` 构建同题对照与隔离盲测场景，重点考察在给定 Rachel 上下文的前提下，LLM 的选择质量、选择稳定性以及是否存在超越模板的增益。为避免提示泄露和数据污染，E2 强调 prompts 与 answers 的物理隔离，并使用 same-question 设计比较 full 与 compact 两种上下文版本。

E2 主要报告 `llm_exact_rate`、`llm_partial_rate`、`llm_beyond_template.rate`、`selection_accuracy.rate` 以及 `llm_choice_dist` 等指标。这里 `selection_accuracy` 衡量的是当模板已经覆盖正确答案时，LLM 是否能够选中正确候选；而 `llm_beyond_template` 则衡量当模板未能命中时，LLM 是否能够通过 custom、bond alternative、FGI 或其他形式提供额外增益。这样的指标设计体现了 Rachel 对 LLM 角色的理解：LLM 的价值不一定体现在“整体替代模板”，而更可能体现在“在结构化动作空间中提升选择与组织质量”。

E3 则进一步追问：如果 LLM 的确体现出 beyond-template 增益，这种增益究竟来自什么机制。为此，我们对 E2 中的 `template_miss & llm_hit` 子集进行机制标注，将其划分为 `custom`、`bond-alt`、`fgi`、`smart_cap` 等类别，并对每类抽取代表案例。E3 的意义在于，它把“LLM 有帮助”从一个粗粒度统计现象，进一步落实为可解释的系统级机制，从而避免将增益错误归因于偶然命中或提示工程运气。

### 5. E4：多步端到端评估

E4 是本文多步能力的主实验，对应本文关于 practical multi-step planning 的核心 claim。与只展示少量精挑案例不同，E4 的目标是在统一协议、统一预算和统一数据池下评估 Rachel 是否能够完成多步规划，并系统记录成功、部分成功与失败的结构。为兼顾横向可比性与真实使用价值，E4 采用双轨设置：其一是 Static comparability track，用于在固定路线池上进行横向可比评估；其二是同上下文会话驱动的主评轨，用于更真实地反映 Rachel 作为交互式多步系统的实际行为。

在数据层面，E4-v2 基于 PaRoutes `n1/n5` 路线资产构建固定会话池，并将其进一步拆分为 dev/eval 子集。所有主评轨实验均在冻结的数据池、固定模型版本、固定预算参数和统一导出路径下运行。具体预算通常记录为 `max_depth`、`max_steps`、`max_nodes`、terminal 判定阈值与模型元数据，以避免不同实验批次之间的隐式条件漂移。该设计的核心目的，是将多步实验从“展示某几个成功案例”推进为“对固定目标池进行可追踪、可复核的系统性评测”。

E4 的结果不仅记录 `complete / partial / fail` 等工程标签，还记录步数、深度、节点数、terminal 数量、feasibility 分布以及与参考路线的若干对齐指标。更重要的是，本文在 E4 中明确区分两种成功概念：其一是路径是否完成并得到化学上可接受的终端原料；其二是路线是否与参考路线在关键边结构上保持一致。前者更接近 `practical success`，后者更接近 `reference-route similarity`。我们认为，在多步规划中，如果只报告后者而忽视前者，就会错误地把“没有重建参考答案”直接等同于“系统无效”；而 Rachel 的目标恰恰是证明这种等同并不成立。

### 6. E5：必要消融

E5 用于量化 Rachel 中几个关键设计成分各自的贡献，包括 bond-only、bond + FGI、bond + FGI + smart_cap 以及不同提示信息量条件。该实验的意义不在于展示“我们也做了消融”，而在于支撑一个更强的系统论点：Rachel 不是一个不可分割的黑箱整体，而是由多个有明确功能边界的设计组件组成，这些组件的贡献应当能够被分别识别、量化与比较。

具体而言，bond-only 条件可以帮助回答“若仅保留断键搜索，系统会损失多少能力”；加入 FGI 后的条件则衡量官能团互变作为主算子的实际增益；加入 smart_cap 的条件则检验局部环境规则是否有助于越过模板覆盖边界；在此基础上，full prompt 与 compact prompt 的对照又可以揭示上下文组织方式对策略层选择质量的影响。这样的消融结构与 Rachel 的设计层次高度一致，因此不仅有实验意义，也有叙事意义：它让文章能够明确回答“Rachel 的每一个核心设计是否真的值得存在”。

### 7. E6：失败模式与边界

E6 对应 Rachel 作为形式化化学推理系统的另一项关键主张：系统的失败不是不可描述的黑箱误差，而是可被分类、统计和治理的结构性对象。该实验综合使用 E4 的失败样本、E2 的 miss 案例以及 baseline 结果分析，对失败模式进行 taxonomy 划分，并生成结构化案例与可视化诊断页面。其目的不是仅仅列举几个失败例子，而是识别系统在模板质量、候选生成、搜索偏置与验证门之间的主要张力。

这种失败分析对于 Rachel 尤其重要，因为 Rachel 的系统价值并不只体现在成功样本上，还体现在它是否把错误暴露为系统可见对象。如果一个逆合成系统的失败只能以“模型没答对”收场，那么它很难支持真正的改进闭环；但如果失败能够被分解为 validation gate、depth limit、operator bias、template mismatch 等可统计类别，那么系统就真正具备了治理和演化的基础。

### 8. 评测指标

为了匹配 Rachel 的多层目标，本文采用多层次指标体系，而不是单一准确率。对单步任务，我们同时报告精确匹配率（exact match）、部分匹配率（partial match）和 Tanimoto 相似度分布。精确匹配率对应严格的前体集合匹配，部分匹配率反映候选是否捕捉到部分真实反应结构，而 Tanimoto 则提供连续的结构相关性度量。对多步任务，我们同时报告工程结局标签、可行性相关指标和路线对齐指标。工程结局用于衡量系统是否在预算内完成规划；可行性指标反映终端可接受性与局部步骤的验证质量；路线对齐指标则用于衡量与参考路线之间的相似程度。

特别地，在多步场景中，我们建议将质量评价分为两层。一层是实用成功（`practical success`），即路线是否在给定预算下得到化学上可接受的终止结果；另一层是参考路线相似度（`reference-route similarity`），即其关键步骤、长度和边结构与参考路线的一致程度。前者对应系统作为规划工具的现实价值，后者对应系统作为 benchmark 参与者的横向可比性。Rachel 的实验设置刻意将两者并列报告，正是为了避免把多步系统的价值压缩为单一重建指标。

### 9. 统计协议与显著性报告

本文尽量避免只给出点估计而不提供统计区间。对于比例类指标，如精确匹配率、部分匹配率、成功率和超模板命中率（beyond-template rate），我们采用 Wilson 区间报告 95% 置信区间。对于连续型指标，如 Tanimoto、可行性得分（feasibility score）或若干路线长度偏差统计，我们采用 bootstrap 方法估计 95% 置信区间。在条件对比方面，对于比例差异采用 Fisher 或同类比例差异检验，对于连续指标分布差异采用 Mann–Whitney U 检验，并辅以 Cliff’s delta 或根据分布条件选用适当效应量。

采用上述统计协议有两层原因。第一，它能使 Rachel 的实验结论避免停留在经验性印象上，而更接近可复核的定量论证。第二，它与 Rachel 强调“过程可验证”的方法论保持一致：既然系统本身要求每一步有证据与检查，那么实验结论也应尽量以区间、显著性和效应量的形式呈现，而不是只给出一组单点数字。

### 10. 复现、公平性与实验冻结

由于 Rachel 涉及多步会话、不同模型设置、不同提示版本和多个实验轨道，实验复现与公平性控制尤为重要。为此，本文遵循一套冻结原则：固定数据池、固定随机种子、固定模型版本与变体、固定预算上限、固定 terminal 判定规则，并将所有实验产物写入标准化结果目录。对于 E4 主评轨，我们进一步要求 A/B 双轨使用同一目标池、同一预算和同一分层口径，并为每个 target 保留唯一的 run 记录与日志文件路径。对于 E2，则要求 prompts 与 answers 物理隔离，以避免任何形式的数据泄露或人工后见污染。

这一复现协议并不是附属性的工程习惯，而是 Rachel 论文叙事的重要组成部分。因为 Rachel 所主张的并不是“让 LLM 更自由地发挥”，而是“让多步逆合成成为一个更可控、更可检查的系统过程”。如果实验本身缺少冻结和记录，那么文章在方法上强调的形式化与可审计性就无法在评测层面闭环。

### 11. 本节小结

总体而言，Rachel 的实验设置遵循与其系统设计一致的基本原则：分层提出问题，分层组织证据，分层报告结论。E1 保证动作空间底盘的质量，E2/E3 解释策略层的作用，E4 评估多步 utility，E5 拆解系统组件贡献，E6 则把失败转化为可治理对象。这样的实验设计不是为了让文章“显得实验很多”，而是为了让 Rachel 的核心主张——多步逆合成应被组织为一种形式化化学推理过程——在方法、评测和结论三个层面形成一致闭环。

## 讨论

与结果部分不同，本节的目标不是继续罗列数字，而是解释 Rachel 作为一个系统范式，究竟为多步逆合成带来了什么新的表述方式。由于当前实验尚未全量冻结，本节只保留**弱依赖实验结果的概念性讨论**；凡是直接依赖实验对比、定量结论或 failure taxonomy 分布的判断，均改写为统一的实验回填占位。换言之，本节所要回答的不是“Rachel 最终能排到第几”，而是“Rachel 为什么值得被当作一种新的多步逆合成组织方式来讨论”；而所有经验性支持，将在实验完整冻结后按统一模板回填。

### 1. Rachel 重新定义的不是一个模块，而是问题表述

本文最希望传达的一点，是 Rachel 的核心贡献并不只是一个更强的候选生成器，也不只是把 LLM 接到若干化学工具之后形成一个可交互系统。更本质地说，Rachel 试图重新定义“多步逆合成究竟是什么问题”。在传统表述中，多步逆合成常被拆成若干相互连接的单步预测，或被视为一类特殊的树搜索问题；而在 Rachel 的表述中，它首先是一个**状态—动作—验证—提交**的闭环推理任务。这个转变看似只是工程组织方式的变化，实际上却改变了整个系统被理解、被评测、被优化的方式。

这种问题表述上的提升，是 Rachel 最接近“形式化化学推理”的地方。这里所谓“形式化”，并不是声称我们已经给出了化学合成的完备公理系统，也不是要把系统包装成严格意义上的定理证明器（theorem prover）；更准确地说，Rachel 提供的是一种**操作性的形式化**：分子状态被显式表示，合法动作被离散化，动作结果必须接受验证门检查，正式提交会留下可追溯证据，而失败也不再是消失在自然语言中的无名噪声。正因为有了这一层操作性形式化（operational formalization），我们才有可能把多步逆合成从“模型给出一个看起来合理的答案”推进到“系统按协议完成一段可检查的推理过程”。

如果从论文叙事角度看，这一点尤为重要。因为 Rachel 真正值得被“卖”的，不是某一个组件是否又比已有方法多了几个百分点，而是它把研究对象从“反应建议”提升成了“推理协议”。这使得文章天然可以从更高抽象层展开：我们不再只讨论哪个模型更会猜，而是讨论什么样的系统组织方式更适合承载复杂、多约束、可失败、可修复的化学规划任务。

### 2. 化学工具与 LLM 的分工，应当是“事实层—策略层”而非“弱模型—强模型”

Rachel 的另一个关键启示在于，它提供了一种更合理的人机/模组分工方式。当前不少 LLM 化学系统的隐含设定是：只要模型足够强，它就可以逐渐吞并模板、规则和验证模块的职责；而 Rachel 的结果与实现逻辑都指向另一种更稳健的路线，即**化学工具负责事实层，LLM 负责策略层**。工具层提供结构分析、模板执行、FGI 变换、正向验证、原子守恒与环境兼容性检查；LLM 则基于这些结构化事实进行候选比较、优先级排序、局部试探和失败后再规划。

这一分工并不只是工程上“好维护”而已，它还直接决定了系统能否可控。若把化学正确性也交给自然语言模型隐式承担，那么系统一旦失败，很难区分是知识错误、搜索错误、解析错误还是验证错误；相反，在 Rachel 中，错误发生在哪一层、为什么发生、能否通过调整模板或先验修正，都可以被相对清晰地定位。换句话说，Rachel 不是在追求一个“化学全能模型”，而是在构造一个**能让智能与约束协同工作**的系统。对多步逆合成这种高约束任务而言，这种分工很可能比“让一个大模型单独包办一切”更现实，也更容易持续迭代。

这一分工还有一个重要写作优势：由于 Rachel 不是一个不可分解黑箱，而是一个职责清晰、证据分层的系统，因此最终论文的说服力不必全部压在单一总榜表格上。实验全量完成后，工具层、策略层、多步层与失败治理层可以分别提供证据，并共同支撑同一条主叙事。

### 3. FGI 的意义在于改变可达性，而不是补模板

### 6. 局限性：Rachel 仍然是一个“强约束但不完备”的系统

为了保持论文论证的可信度，讨论部分也必须主动承认 Rachel 当前的边界。首先，Rachel 的形式化依然是一种**工程意义上的形式化**，而不是从化学第一性原理推导出的完备形式系统。它的动作空间、验证规则和模板资产都带有明显的人类设计痕迹，因此系统能力上限仍然受模板覆盖、规则粒度与验证器保守性的共同制约。其次，即便最终实验将进一步支持 FGI 的重要性，FGI 模板本身的质量仍未必均衡，某些 prereq SMARTS 过宽、某些位点选择性不足，这意味着“FGI 很重要”并不等于“FGI 已经做好了”。

第三，Rachel 当前对路线质量的建模仍然偏结构性，而非工艺性。系统尚未系统性整合试剂成本、条件鲁棒性、收率分布、保护基经济性、放大可行性等更接近真实工艺化学的目标函数。第四，虽然会话机制和审计状态使其非常适合人机协同，但当前论文中的主评轨结果仍未完全冻结，因此关于“结构化会话究竟比 static pipeline 强多少”的最终经验判断仍应保持克制。最后，Rachel 当前仍在一定程度上依赖人为设定的 operator prior，这意味着系统还未完全实现针对不同目标分子族的自适应动作调度。

这些局限并不会削弱本文的主张，反而有助于把 Rachel 放在一个更准确的位置上：它不是终点，不是一个已经解决多步逆合成的全能系统，而是一个把问题重新组织得更可研究、更可改进、更可审计的平台。

### 7. 未来方向：从“可运行的形式化”走向“可学习的形式化”

基于以上讨论，Rachel 最自然的后续方向并不是简单地继续堆模板或更换更大的语言模型，而是推进“可学习的形式化化学推理”。第一条路线是让算子先验（operator prior）变得自适应：不同分子、不同官能团环境、不同路线深度下，断键与 FGI 的优先级未必应当固定，而可以通过历史成功率、局部验证信号或策略学习动态调整。第二条路线是让验证门从静态规则扩展为更强的多级验证体系，例如把快速结构验证、局部正向验证与更高代价的反应可行性评估结合起来，实现分层筛选。

第三条路线是把 failure taxonomy 真正接入训练和搜索回路，使失败不只是报告结果，而是成为策略更新的监督来源。第四条路线是将路线评估从单一结构匹配扩展为多目标规划，在 `practical success` 与 `reference-route similarity` 之外纳入长度、收敛性、可采购性、风险与复杂度等指标。第五条路线则是强化人机协同：既然 Rachel 已经具备命令式会话与审计状态，那么它天然适合引入化学家反馈、偏好修正和交互式约束编辑，从而把系统从基准测试智能体（benchmark agent）进一步推进为真正可用的合成规划助手。

如果用一句话概括这些未来方向，那就是：Rachel 的长远价值，不只是在于“它已经证明了什么”，更在于“它使哪些问题第一次能够被清楚地提出、被结构化地研究、被系统性地改进”。
